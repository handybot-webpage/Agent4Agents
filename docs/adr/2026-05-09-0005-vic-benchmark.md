# ADR 0005 — Vehicle Interior Cleaning benchmark scaffold + Isaac Sim selection

- **Status:** Accepted
- **Date:** 2026-05-09
- **Supersedes:** —
- **Superseded by:** —

## Context

The project is exercising Project Compact + gstack on a real, non-trivial domain task to validate that the framework holds up under load (not just framework infra changes). The chosen domain task is a robotics simulation benchmark: Vehicle Interior Cleaning (VIC).

VIC is *paper-inspired by* but *domain-shifted from*
[CleanUpBench (arxiv 2508.05543)](https://arxiv.org/abs/2508.05543) — Wenbo
Li et al, an indoor sweeping+grasping benchmark for mobile manipulators on
NVIDIA Isaac Sim. Our domain shift is from indoor rooms to vehicle
interiors: smaller enclosed space, occluded geometry, fabric surfaces,
fragile components (mirrors, dashboard), vehicle-specific clutter classes
(sand from floor mats, food crumbs, child toys, drink containers).

Two coupled architectural choices need to land in this ADR:

1. **Should this project host a benchmark module at all?** (scope question)
2. **Which simulator should it target?** (irreversibility-list dependency)

### Considered simulators
- **NVIDIA Isaac Sim.** Matches CleanUpBench. Photorealistic. Mature mobile manipulation. Isaac Lab gives ready RL training pipelines. **Linux-first; macOS unsupported; requires NVIDIA RTX GPU.** 10+ GB install.
- **MuJoCo.** Native macOS / Apple Silicon. `pip install mujoco` one-liner. Industry-standard for contact-rich manipulation. Open source. **Less photorealism**, no off-the-shelf vehicle assets, departs from source paper's choice.
- **Newton (NVIDIA).** Recent differentiable physics engine. Cutting-edge but immature for contact-rich manipulation; small ecosystem; thin tutorials. Risk of unbounded yak-shaving.

The choice was surfaced to the project owner via `AskUserQuestion` per the irreversibility protocol in `agent-context/03-question-protocol.md`. The owner chose Isaac Sim, accepting the macOS-incompatibility tradeoff for paper-faithfulness and mature RL pipelines.

## Decision

### 1. Project hosts both governance infra AND domain benchmarks
The Agent4Agents repository is now a multi-purpose project:
- **Top-level:** Project Compact governance substrate (`agent-context/`, `docs/adr/`, `tasks/`, `bin/compact-check`, etc.). Stable.
- **`benchmarks/`:** New top-level directory hosting domain benchmarks. The first is `vehicle-interior-cleaning/`; future benchmarks may follow the same template.

Each benchmark is a self-contained module with its own MODULE.md, configs, and tests. Adding a new benchmark is a Tier 2 change with a new ADR, never silent.

### 2. Sim engine: NVIDIA Isaac Sim (with Isaac Lab)
The VIC benchmark targets **Isaac Sim 4.x** with **Isaac Lab** for the RL/eval API, on a **Linux x86_64 + NVIDIA RTX** host.

- Python package: `isaaclab` (and underlying `omni.isaac.*`).
- Asset format: USD.
- Physics: PhysX (Isaac Sim default).

### 3. Scaffold extends, doesn't reproduce
The scaffold delivers the *correct shape* of a benchmark — SPEC, env, tasks, metrics, baselines, eval runner, configs — with stubs where Isaac Sim is required but unavailable on the developer's machine. Pure-Python parts (metric math, config parsing, task definitions) are runnable on macOS for fast iteration; Isaac-Sim-dependent parts gracefully degrade with explanatory ImportErrors.

The scaffold does **not** include trained RL policies, a leaderboard, real vehicle USD assets, or numerical reproduction of CleanUpBench results.

## Consequences

**Positive**
- Paper-faithfulness: same simulator, same skill primitives (sweep + grasp), same general metric categories.
- Future RL baselines can plug into Isaac Lab's existing PPO/SAC trainers without re-architecting.
- Sets a precedent that benchmark modules live under `benchmarks/<name>/` with the same `MODULE.md` discipline as any other module.
- Photorealism preserves the option of vision-policy work later.

**Negative**
- **macOS dev loop is degraded:** the developer cannot run end-to-end simulation on this machine. Iteration on env code requires either a Linux box, a cloud GPU instance, or a Docker GPU passthrough.
- Isaac Sim's install footprint and dependencies are heavyweight — anyone reproducing the benchmark needs >50 GB free, an NVIDIA RTX GPU, and an Ubuntu-class Linux distro.
- Coupling to NVIDIA's stack: if Isaac Sim's API changes meaningfully, the env code requires porting. Mitigation: thin wrapper around Isaac Lab's `ManagerBasedRLEnv`; benchmark-specific code is sim-agnostic where feasible.
- This decision is on the project's irreversibility list (heavy dependency add). Switching sim engines later requires a superseding ADR + porting all asset/env code.

**Mitigation: graceful import degradation**
The Python package is structured so that `import vic` succeeds on machines without Isaac Sim. Sim-dependent symbols raise a clear, actionable `ImportError` only when actually invoked. This lets unit tests and metric-shape checks run on any developer machine, including macOS.

**Mitigation: documented setup**
`benchmarks/vehicle-interior-cleaning/README.md` documents the platform requirements up front (Linux x86_64, NVIDIA RTX 30/40-series or better, ≥16 GB VRAM, Ubuntu 22.04+, Isaac Sim 4.x, Isaac Lab 2.x). The reader knows on line 1 whether their machine can run this.

## Alternatives considered (rejected)

- **MuJoCo.** Would have allowed full local execution on the developer's macOS Apple Silicon. Better dev loop. Industry-standard for contact-rich tasks. Rejected by the project owner because it departs from CleanUpBench's choice and weakens the photorealism story for future vision-policy baselines. Tradeoff explicitly acknowledged.
- **Newton.** Cutting-edge but ecosystem too thin for shipping a working scaffold without weeks of integration work. Rejected as too risky for the charter's scope.
- **Skip the benchmark; demonstrate framework on a smaller task.** Considered. Rejected because the user's instruction was explicit (vehicle interior cleaning, paper-referenced, sim-engine choice given). Substituting a smaller task would defeat the test.

## What this ADR does NOT decide

Deferred to future ADRs / charters as the benchmark matures:

- The exact Isaac Lab API version pin (will pin in `pyproject.toml` once the host machine is provisioned).
- Whether to support headless mode for CI evaluation.
- The leaderboard / evaluation-server format (out of scope for the scaffold).
- Asset licensing (will need an ADR if real vehicle CAD models are imported).
- RL training infrastructure (separate Tier 2 charter when needed).
